# Milestones: локальный AI-наблюдатель

## Цель MVP

Собрать полностью локальное foreground-приложение для iOS и Android, которое:

- постоянно анализирует изображение с камеры;
- формирует устойчивое состояние сцены вместо обработки каждого кадра;
- каждые 10 секунд генерирует свободный комментарий о происходящем;
- озвучивает комментарии;
- после фразы «Ассистент» слушает вопрос и отвечает с учётом текущей сцены.

Фото, видео, аудио, история комнаты и диалоги между запусками не сохраняются.
На диске остаются только скачанные модели. Первый контрольный iOS-девайс —
iPhone 11 с 4 ГБ RAM; Android acceptance выполняется на 64-битном ARM
Android Emulator. Deployment target проекта остаётся iOS 13, Android min SDK —
24.

## Общие требования

- Все ML-операции выполняются локально. Интернет нужен только для первой
  загрузки моделей.
- Камера, YOLO и голосовой агент работают только пока приложение находится
  в foreground.
- Каждый runtime скрывается за application-портом; типы Flutter-плагинов,
  FFI и native API не выходят из `data/` или app-level composition.
- Асинхронные операции имеют явно определённую concurrency policy и корректно
  освобождают камеры, модели, таймеры, подписки и аудиоресурсы.
- Каждый milestone оформляется отдельным Conventional Commit.

## Quality gate после каждого milestone

Milestone не считается завершённым, пока не пройдены статическая проверка и
семантический review Flutter Agentic Harness.

Обязательная последовательность:

1. Завершить реализацию и ближайшие полезные тесты milestone.
2. Выполнить static Harness check:
   `dart run tool/harness.dart verify --changed`.
3. Запустить skill `$harness-review` для полного diff текущего milestone.
4. Если review нашёл verified findings уровня `blocker` или `major`, запустить
   цикл:
   `Fix → Static Harness check → $harness-review again`.
5. Повторять цикл, пока `$harness-review` не вернёт ни одного verified
   `blocker` или `major`.
6. Только после закрытия блокирующих findings milestone можно коммитить и
   переходить к следующему.

Исправления из review должны сопровождаться ближайшими регрессионными тестами,
когда finding описывает проверяемое поведение. Нельзя ослаблять architecture
checks, quality checks или baseline ради прохождения цикла.

## 1. Live YOLO

Заменить существующую передачу сырых кадров из
[camera feature](lib/features/camera) на нативный `YOLOView` из
`ultralytics_yolo`.

### Реализация

- Подключить `ultralytics_yolo ^0.6.10`.
- Использовать модель `yolo26n`, разрешение 720p, confidence `0.4`,
  IoU `0.7` и GPU с нативным CPU fallback.
- Показывать нативные bounding boxes, классы и confidence.
- Вывести диагностические FPS и время inference.
- Поддержать прогресс первой загрузки модели, retry и cached offline launch.
- Сохранить смену камеры и lifecycle pause/resume.
- Преобразовывать `YOLOResult` в собственные domain-типы на границе адаптера.

### Готово, когда

- YOLO непрерывно работает на iPhone 11 не менее 5 минут без падений.
- Повторный запуск с уже скачанной моделью работает без интернета.
- Средний FPS на контрольном устройстве не ниже 15.
- Смена вкладки, background/foreground и смена камеры не оставляют занятую
  camera session.

## 2. Устойчивая модель сцены

Преобразовать шумный поток YOLO callbacks в устойчивое состояние комнаты.

### Реализация

- Добавить чистые domain-типы `Detection`, `NormalizedBox`,
  `TrackedObject` и `SceneSnapshot`.
- Считать объект устойчивым, если он присутствует минимум в 3 из последних
  5 кадров.
- Сопоставлять объекты одного класса при IoU не ниже `0.5`.
- Считать объект исчезнувшим после его отсутствия в 3 из последних 5 кадров.
- Выдавать объектам session-scoped ID.
- Определять положение по сетке 3×3: `leftTop`, `top`, `rightTop`,
  `left`, `center`, `right`, `leftBottom`, `bottom`, `rightBottom`.
- Публиковать новое состояние только при реальном изменении устойчивой сцены.
- Не добавлять постоянное хранилище.

### Готово, когда

- Единичные пропуски YOLO не вызывают мерцание объектов.
- Повторные detections одного предмета сохраняют один ID.
- Появление, перемещение и исчезновение детерминированно покрыты unit-тестами.

## 3. Qwen3-0.6B, нативный llama.cpp и менеджер моделей

Добавить локальную текстовую модель и ручной диалог независимо от
автоматического наблюдателя.

### Нативный runtime

- Подключить `ggml-org/llama.cpp` как git submodule, зафиксированный на точном
  upstream commit.
- Собирать `llama.cpp` и GGML собственным Dart build hook для iOS device,
  Simulator и Android. iOS использует Metal с CPU fallback и deployment target
  iOS 13; Android собирает переносимый CPU backend через зафиксированный NDK.
- Не скачивать исходники или готовые native binaries во время сборки
  приложения: артефакт должен воспроизводиться из pinned submodule и
  зафиксированной build-конфигурации.
- Использовать собственную узкую FFI-границу для загрузки и выгрузки модели,
  streaming tokens, cancel и освобождения native-ресурсов. Вызовы inference
  не выполняются на UI isolate.
- Не подключать pub.dev-обёртки над `llama.cpp` и не поднимать локальный HTTP-
  сервер внутри приложения.
- Зафиксировать upstream commit, build flags и MIT-лицензию `llama.cpp` в
  документации third-party компонентов.

### Модель

- Модель: `Qwen3-0.6B`.
- Квантизация: `Q4_K_M`.
- Источник: `unsloth/Qwen3-0.6B-GGUF`.
- Ориентировочный размер GGUF: 397 МБ.
- Контекст MVP: 2048 токенов.
- Автоматического перехода на Q3 или другую модель нет.

### Реализация

- Добавить `ModelStore` со статусами download, verify, ready, loading,
  failure и retry.
- Зафиксировать в model manifest URL конкретной ревизии, размер, SHA-256,
  имя файла и лицензию Apache-2.0.
- До загрузки проверять свободное место; после загрузки проверять checksum.
- Повторно использовать только полностью проверенный cached-файл.
- Добавить application-порт `CommentGenerator`.
- На экране
  [assistant page](lib/features/assistant/presentation/pages/assistant_page.dart)
  реализовать ручной prompt, streaming-ответ, cancel и retry.
- Для обычных комментариев использовать `/no_think`, `maxTokens: 40`,
  temperature `0.7`, top-p `0.8`, top-k `20`, публиковать первое законченное
  английское предложение не менее чем из трех слов и после него останавливать
  native generation.
- Для ручного диалога разрешить `/think` с общим лимитом
  `maxTokens: 512`.
- Не показывать `<think>` пользователю и не возвращать reasoning в историю.

### Мультимодальная граница

- Нативная build-конфигурация и FFI ownership не должны предполагать, что
  runtime навсегда останется text-only: будущий visual adapter использует тот
  же pinned `llama.cpp` и `libmtmd` с отдельным `mmproj`.
- Выбор VLM, получение camera keyframe, загрузка `mmproj` и image inference не
  входят в этот milestone. Они оформляются отдельным milestone после model и
  hardware spike, а не добавляются скрыто в текстовый runtime.

### Готово, когда

- Qwen загружается одновременно с активным YOLO на iPhone 11 без memory
  termination.
- Модель стабильно отвечает на английском языке.
- Автоматический короткий комментарий генерируется не дольше 10 секунд.
- Десятиминутный тест повторных генераций не приводит к падению или
  неконтролируемому росту памяти.
- Модель повторно запускается из кэша без интернета.
- Device smoke подтверждает streaming, cancel, unload и повторную загрузку
  через собранную собственным build hook native-библиотеку.
- Android emulator smoke подтверждает те же операции, а также повторный запуск
  из проверенного кэша при отключённом transport.

Если iPhone 11 не проходит memory smoke, повышается минимальный класс
поддерживаемого устройства; выбранная модель не заменяется.

## 4. Автоматический AI-наблюдатель

Соединить устойчивую сцену и Qwen в непрерывную foreground-сессию.

### Реализация

- Добавить `ObserverBloc`, владеющий таймером, текущей сценой, LLM-запросом
  и session transcript.
- Интервал по умолчанию — 10 секунд.
- Доступные интервалы: 10, 30, 60, 120 и 300 секунд.
- Настройка интервала действует только до закрытия приложения.
- Передавать модели актуальную сцену, предыдущий комментарий и последние
  четыре диалоговые пары.
- Разрешить свободный разговорный комментарий и интерпретацию сцены.
- Использовать policy `ignore`: если предыдущая генерация ещё выполняется,
  очередной tick пропускается.
- Показывать streaming-текст, объекты сцены и состояние модели.

### Готово, когда

- Наблюдатель работает не менее 10 минут без перекрывающихся генераций.
- Каждый новый запрос использует последнюю доступную устойчивую сцену.
- Остановка наблюдения отменяет таймер и активную генерацию.

## 5. Системный Text-to-Speech

Добавить первый рабочий голос без отдельной TTS-модели.

### Реализация

- Добавить application-порт `SpeechSynthesizer`.
- Реализовать адаптер на `flutter_tts` с локалью `en-US`.
- Озвучивать только завершённый комментарий.
- Добавить mute, stop и replay.
- Во время генерации или речи пропускать очередной автоматический tick.
- Не создавать очередь устаревших комментариев.

### Готово, когда

- Каждый завершённый комментарий можно воспроизвести, остановить и повторить.
- Отключение звука не останавливает наблюдение и генерацию текста.
- Lifecycle и audio interruptions не оставляют зависший speech session.

## 6. Локальная Piper TTS-модель

Заменить зависимость от установленного системного голоса собственной
офлайн-моделью.

### Реализация

- Подключить `sherpa_onnx` и выполнять синхронный native synthesis вне UI
  isolate.
- Зафиксировать English US модель
  `vits-piper-en_US-ljspeech-medium-int8`: один speaker, 22 050 Гц,
  ориентировочно 21 МБ в архиве, 38 МБ во временном развёрнутом tar и 37 МБ
  после извлечения.
- Загружать модель через общий типизированный `ModelStore`: проверять размер и
  SHA-256 архива, а затем размер, количество файлов и canonical SHA-256 всего
  извлечённого дерева до атомарной публикации кэша.
- Хранить manifest, storage reserve и параметры sherpa/VITS
  (`provider`, threads, speaker, noise/length scales, speed, silence и sentence
  limit) в compile-time `.env`.
- Учитывать точный размер временного tar в storage preflight и гарантированно
  удалять его после успешной или неуспешной попытки извлечения.
- Сделать Piper основной реализацией `SpeechSynthesizer`.
- Оставить системный TTS техническим fallback только для ошибки до начала
  локального воспроизведения; не повторять уже начатую реплику другим voice
  backend.
- Освобождать Piper runtime до воспроизведения, чтобы не держать его в RAM
  одновременно с YOLO и Qwen без необходимости.
- Передавать сгенерированный PCM в in-memory playback и не сохранять аудио
  реплики на диск.

Статус: завершено. iOS Simulator, Android Emulator и iPhone 11 hardware
acceptance проходят полный production-path сценарий.

### Готово, когда

- English речь синтезируется полностью офлайн.
- Повторное воспроизведение не требует повторного скачивания модели.
- Одновременная работа YOLO, Qwen и синтеза проходит hardware smoke на
  минимально поддерживаемом устройстве.

## 7. Hands-free агент

Добавить голосовое пробуждение, распознавание вопроса и голосовой ответ.

### Реализация

- Добавить разрешение микрофона и состояния:
  `watching → wakeDetected → listening → thinking → speaking`.
- Использовать some model для streaming English ASR; размер модели около 138 МБ.
- Определять wake phrase «Ассистент» по нормализованной streaming-
  транскрипции.
- После wake phrase слушать до 1.2 секунды тишины, но не дольше 15 секунд.
- Передавать Qwen вопрос, текущую сцену и последние четыре диалоговые пары.
- Отключать распознавание во время TTS, чтобы агент не слышал собственный
  ответ.
- Не поддерживать barge-in в MVP.
- Не сохранять аудио и транскрипты на диск.

### Готово, когда

- Wake phrase запускает одну listening session без повторных срабатываний.
- После окончания речи вопрос передаётся Qwen ровно один раз.
- Ответ учитывает текущую сцену и озвучивается.
- Отказ в разрешении, тишина, пустая транскрипция и ошибка ASR имеют
  восстановимые UI-состояния.

## 8. Стабилизация iOS и Android MVP

Подготовить полный стек к длительной работе на реальном устройстве.

### Реализация

- Объединить загрузку YOLO, Qwen, Piper и ASR в setup-экран.
- Перед полной установкой моделей требовать минимум 1.5 ГБ свободного места.
- При memory warning останавливать сессию, освобождать модели и предлагать
  restart.
- При thermal state `serious` отключать автоматические комментарии, сохраняя
  ручной запрос.
- При thermal state `critical` останавливать всю AI-сессию.
- Добавить русскую локализацию интерфейса.
- Зафиксировать AGPL-3.0 для YOLO как допустимое ограничение открытого
  pet-проекта.

### Готово, когда

- Пройден 30-минутный soak-тест: камера, YOLO, периодические комментарии,
  TTS и голосовые вопросы.
- Проверены background/foreground, входящий звонок, audio interruption,
  отказ в разрешениях, потеря сети и offline restart.
- Нет утечек camera/audio sessions и неконтролируемого роста RAM.

## Архитектурные контракты

- `YoloObservationAdapter` — data-адаптер, владеющий live YOLO lifecycle и
  преобразованием plugin callbacks в observation events.
- `YoloObservationSurface` — presentation-виджет над `YOLOView`; app-layer
  связывает его с data-адаптером в composition root.
- `SceneStabilizer` — чистая domain-логика стабилизации сцены.
- `ModelStore` — загрузка, checksum и кэширование весов.
- `CommentGenerator` — application-порт локальной LLM.
- Собственный llama.cpp adapter — единственный владелец FFI, model/context
  handles, token callbacks и native lifecycle; `libmtmd` подключается через
  эту же границу в отдельном milestone.
- `SpeechSynthesizer` — общий контракт для system TTS и Piper.
- `SpeechRecognizer` — streaming ASR и endpoint detection.
- `ObserverBloc` — владелец таймера, подписок и AI concurrency policy.

Прямое Cubit-to-Cubit или Bloc-to-Bloc взаимодействие не добавляется.
Связь между feature-срезами выполняется через typed streams, application-
контракты или app-level coordinator.

## Общий test plan

- Unit-тесты IoU, стабилизации 3/5, tracking ID, исчезновения и регионов.
- Repository-boundary тесты YOLO mapping, загрузки моделей, checksum и
  offline cache.
- Native smoke-тесты build hook, загрузки Qwen, streaming, cancel, unload и
  повторного создания llama.cpp runtime на iOS и Android.
- Bloc-тесты таймера, busy ticks, latest scene, lifecycle и cleanup.
- Тесты Qwen prompt template, `/think`, `/no_think`, удаления reasoning из
  истории и token limits.
- Voice-тесты wake phrase, false positives, silence endpoint и блокировки
  ASR во время TTS.
- Интеграционный путь:
  `models → YOLO → stable scene → Qwen → TTS → wake phrase → question → answer`.

## Вне текущего MVP

- Background-наблюдение и background wake word.
- Персистентная память комнаты или диалогов.
- Сохранение фотографий, видео или аудиозаписей.
- Embeddings и vector database.
- Barge-in и прерывание ответа голосом.
- Дообученная YOLO-модель для `clutter`, `dirty cup`, `unmade bed` и других
  классов, отсутствующих в COCO.
